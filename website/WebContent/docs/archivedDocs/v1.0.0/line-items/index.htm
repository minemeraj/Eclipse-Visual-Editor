Visual Editor 1.0.0 Line Item Objectives

The following items need to be designed and implemented for the SWT capability in the Visual Editor. Links are provided where more details and explanations are needed to understand the requirements.

Note: The main objective is to define and iterate through the various line items that it will take to create a SWT extension for the VE framework.  Furthermore; our goal is to create the SWT support in two major stages.  Initial support, and full support.  The initial support will address core platform issues requiring one to support SWT widgets, whereas the full support will come to enhance VE's SWT end user support.  For example, our first goal is to support core SWT widgets (org.eclipse.swt.widgets), and a simple SWT code pattern.  The full support effort will enhance VE to support customized widgets (org.eclipse.swt.widgets.custom), and factory based code patterns etc.

 

Line Item Contribution Process

  1. Review the following line items to get a basic understanding of the requirements and what needs to be implemented
  2. Review the contributors assignment list and those items which have not been assigned.
  3. Look for items of interest in which you would like to develop and contribute.
  4. Ask and submit questions you might have about the line items to the VE development mailing list ve-dev@eclipse.org
  5. Submit your request to the VE development mailing list (ve-dev@eclipse.org) for the specific line item(s) you would like to develop and contribute. Please identify what line item, and contact information (i.e. email, phone, etc.).

 

Line Items

Code Parsing and Generation

  • Simple SWT code generation/parsing pattern
  • Events (inner methods ...) differences from Swing
  • ID (information development) - provide technical documents

Dispose Support for Shared Widgets (e.g. Color)

  • Modeling
  • Code generation

SWT Constructor style bits

  • SWEET - BeanInfo extensions
  • Modeling and code generation
  • PropertySheet - break into families - e.g. Button has style bits that define it as push, check, arrow, toggle, etc..
  • target VM has to cope with widget reconstruction because style bits are set on construction only.

Java Application Launcher

  • Handle both Swing and SWT launching
  • ID (information development) - provide technical documents

Visual Class wizard and main(String[] args) Method Creation

  • Create both AWT/Swing and SWT applications.
  • Customized code generated when main method stub is selected to be generated.
  • Popup on the class to generate a main method
  • Code assist template for main customized for SWT and AWT/Swing applications
  • ID (information development) - provide technical documents

Label providers and cell editors

  • Rectangle
  • Point
  • Color
  • Font
  • Cursor
  • Image
  • Java Object reference - the ability to show and select from a list of visual objects applicable to a specific property setting.

SWT Containers

  • Shell
  • Composite
    • Controls relationship in the model, decoders - target VM - Editparts, layout managers, PS
  • Layout support for Shell, Composite, and Group. This includes layout policies, layout helpers for providing U/I feedback, layout data (PS), and layout switching for:
    • null layout
    • FillLayout
    • RowLayout
    • FormLayout
    • GridLayout
  • Group - PS, edit part, direct edit text
  • TabFolder/TabItem - containment relationship in the model, decoders - target VM, EditParts (page switching, graphical and tree design), PS
  • ToolBar/ToolItem -containment relationship in the model, decoders - target VM, Edit Parts (graphical and tree design).  Allow controls to be dropped onto ToolBar and automatically create ToolItem. What about shared tool item controls...similar problem as shared Swing controls like JButton?
  • Tree/TreeItem - containment relationship in the model, decoders - target VM, EditParts (page switching, graphical and tree design), PS
  • CoolBar/CoolItem - containment relationship in the model, decoders, target VM EditParts (graphical and tree design)
  • Dialog - Color, Directory, File, Font, Message, Print - PS only.

SWT Controls

  • Button - push, checkbox, radio, toggle + PS, edit part direct edit
  • Label - PS, edit part, direct edit
  • Text - PS, edit part, direct edit
  • List - PS
  • Combo - PS
  • ProgressBar - PS
  • Scale - PS
  • Slider - PS
  • Sash - PS

SWT Menus

  • Menu/MenuItem (popup and menubar and cascade).  
    • Model relationship, target VM, edit parts, PS, decoders

Palette Support

  • Re-factor GEF changes - merge in GEF modifications for Eclipse 3.0
  • Ability to open and configure drawers - option to load specific drawers - SWT or Swing
  • Customizable palette - allow users to create custom drawers and controls that can be dropped onto the canvas without using the choose bean dialog.
  • Ability to store customized palettes - persist custom palettes per user using configuration tools.

Target VM

  • Changes to handle evaluating initStrings to deal with arguments such as display/composite
  • Widget reconstruction when style bits change because style bits are set on construction only
  • Target VM managers for handling tabfolder, toolbar, menubar, coolbar
    • Needed to handle when the adding, removing, or reordering of items occurs in the remote VM

Context Menu

  • Eclipse plugin  popup extensions
  • ObjectActionDelegates
  • ActionFilters
