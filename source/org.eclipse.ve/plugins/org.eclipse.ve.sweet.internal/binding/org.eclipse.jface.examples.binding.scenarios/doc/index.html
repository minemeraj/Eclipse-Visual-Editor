Data Binding Proposal

Introduction

Displaying and editing model data in a UI usually consists of the following steps:
  1. Creating the UI widget that displays the data and optionally makes it editable.
  2. If the data can be edited in the UI, attaching a listener to the UI widget that validates the change and updates the model data at appropriate times.
  3. If the model data can change through other means while it is displayed in the UI, attaching a listener to the model object and updating the UI widget upon model changes.
  4. Initializing the UI widget's state based on the current model state
The data binding framework addresses steps 2 to 4. It abstracts from the details of attaching listeners to both the UI and the model sides for updating the other side's state by offering an API that allows to bind a UI widget to model data.

Goals

This is a list of goals that we would like to achieve with the data binding framework:
  • Easy to use
    • Binding should be one line of code, easy to read/understand, and CodeAssist-friendly
    • Consistent usage across controls / data types
    • Should not require new or custom controls
    • Ability to designate nested binding (e.g., person.shippingAddress.Street)
  • Easily Extensible / Configurable
    • Extendible for custom controls, data types (event models, derived properties etc.)
    • Bind policies (on the binding service)
    • Configurable with validation, and data conversion, traceability.
  • Separation of concerns
    • Creating widgets
    • Binding
    • Conversion
    • Validation
  • Lifecycle management
    • Single management across the GUI (multiple binds to the same object, really is bound to the same object)
    • Cleanup (e.g., Listeners) should be done automatically
  • Validation
    • At the property level
    • At object level
    • Across multiple Objects
  • Conversion
    • Basic conversion need be available
    • Simple data types from/to String
    • Monetary amounts ($ 1,096.23)
    • Time/date conversion (locale, time zone)
  • Support for nesting
    • The simplest form of binding, is to bind properties of two instances (e.g., TextField/text, and Person/firstName)., where changes to properties on one end, reflects to the other. Another common scenario is that an instance is bounded to an instance that can itself change (e.g., current selected Person instance).
